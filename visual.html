個人がAI相棒の力を引き出すための体系

Fractal Decision Ecosystem

FDEは、AIが本来持っている判断・整理・検証・接続・改善の能力を、毎回ちゃんと発火させるための道具箱です。 会社向けの管理資料ではなく、個人が自分のAIをもっと頼れる相棒にするための入口です。

解きたい問題

AIにはできるはずのことが多いのに、実運用では浅い返答、根拠の混線、危険操作の見落とし、完了の過大申告が起きます。 FDEは、その「発揮されない問題」を先に設計します。

Context 文脈を読めるはずなのに浅い

entryをpacket化し、目的、戻り先、過去文脈へ接続します。

Decision 判断できるはずなのに散らばる

mode、axis、closure_ruleを先に立て、何を決めるturnか固定します。

Evidence 検証できるはずなのに混ざる

事実、推測、不明を分け、根拠のない断定を止めます。

Safety 危険操作を止めきれない

公開、外部送信、visibility変更は人間レビューまで止めます。

Improve 同じ失敗を反省だけで終える

route_failureとして分類し、skill、document、validatorの修正先へ落とします。

Connect 正本がばらける

registryとsource pointerで、正本、外部authority、未吸収candidate、staleを分けます。

FDEの発火順

どこから始まっても、まずこの型へ戻します。これは説明用の飾りではなく、AIに必要な能力を順番に呼び出すための操作手順です。

entryユーザーの言葉から、本当の入口と目的を読む。
packet問い、owner、mode、戻り先、閉じ方を圧縮する。
evidence事実、推測、不明、source pointerを分ける。
decisionadopt / park / discard / decision-needed / blockedを切る。
closure終わったこと、未保証、次の一手を返す。

何で発火させるか

FDEは大きな思想だけではなく、毎回のturnで使える小さな部品の集まりです。

trigger

どの言葉、状態、失敗、操作でFDEを起動するかを決める。

packet

問い、owner、mode、根拠、閉じ方を短く持つ。

8-axis

who / what / why / when / where / how / howmuch / withwhatで迷子化を防ぐ。

gate

公開、外部送信、破壊操作、secret、visibility変更を止める。

registry

詳細手順、外部正本、raw sourceを本文に抱え込まない。

route_failure

能力が発揮されなかった原因と修正先を分類する。

なぜ公開候補にするか

個人で閉じると、到達点はどうしてもその人の得意領域と技量の上限に引っ張られます。 だから、自分が得意なところに尖って取り組み、足りないところはそれを得意な人が補える形にしておく必要があります。

AI格差は生まれます。だからこそ、FDEは「AIをうまく使える人だけの秘密の手順」ではなく、発火条件、判断軸、根拠、公開境界、失敗時の直し方まで見える形にします。

公開境界

このHTMLはローカルで読むための視覚入口です。公開、外部送信、GitHub repository visibility変更は、このHTMLやREADMEだけでは承認されません。

  • README / LICENSE / SECURITY.md / PUBLIC_READY.mdを確認する
  • secret scanとpersonal path scanを確認する
  • draft workspaceを公開境界の外へ置く
  • patent / public kernel gateを確認する
  • 対象repositoryを明示して人間レビューと承認を取る
